# 编码原则

这些规则用于编写和塑造本项目的代码。

它们不是可选的风格偏好，而是构建规则。

## 核心规则

项目代码应该优先追求：

- 权威的集中化
- 执行的模块化
- 用抽象降低复杂度
- 用正交性保持独立性

这四个原则应该同时存在。

集中化：集中定义系统规则。

模块化：模块负责执行系统规则。

抽象：让接口保持简单。

正交化：让独立关注点彼此独立，使变更保持局部。

## 集中化

以下内容应有一个清晰的权威来源：

- 环境变量和配置访问
- schema 和持久数据形状
- 外部服务集成边界
- 路由规则和系统策略
- 其他模块依赖的状态定义

不要在脚本、模块或接口之间复制承重逻辑。

如果系统的多个部分需要同一条规则，应把这条规则移动到一个明确的单一事实来源。

## 面向人，也由机器执行

当一条规则既对人有用，又可以机械检查时，优先同时做到这两点。

面向人的层应该用清晰语言解释规则。

面向机器的层应该在执行成本合理时自动检查规则。

例子：

- 一个任务状态文件可以易读，同时仍包含 `owner`、`next-review` 和 `criterion`
- 一个模块契约可以写明对应测试，而测试可以由项目测试器运行
- commit message 规则可以写在编码标准中，也可以由项目专用检查器验证

当一个小脚本就能执行规则时，不要让承重规则停留在装饰性政策里。

如果判断成本高于它能防止的漂移，也不要过度执行。

## 模块化

把执行拆成有边界的部分：

- 模块
- adapter
- API handler
- UI component
- 维护例程

模块应该易于替换、审计和测试，而不需要重写整个系统。

模块化不等于分散权威。它意味着让执行面保持狭窄、可组合。

## 抽象

把实现复杂度隐藏在稳定、清晰的接口之后。

系统的可见表面应该比背后的机制更简单。

使用抽象是为了减少认知负担，不是制造神秘感。

好的抽象意味着：

- 少量清晰的命令或入口
- 共享工具函数，而不是重复低层逻辑
- 围绕 backend、路径、运行时状态和控制面动作建立稳定接口
- 实现细节留在拥有它的模块里

坏的抽象意味着：

- 模糊的 wrapper 隐藏真正的权威来源
- helper 层复制底层规则
- 命名听起来通用，但没有真正降低复杂度

## 正交化

保持不同关注点彼此独立，使一个变更不会迫使无关地方一起变化。

正交性保护系统不被纠缠。

好的正交性意味着：

- 运行时使用和元开发保持分离
- 策略和执行不坍缩到同一个文件
- 功能启用、backend dispatch、路径解析和审计逻辑有各自边界
- 修改一个 backend 或一个功能，不需要重写无关系统部分

坏的正交性意味着：

- 一个模块承担多个无关职责
- 路径、策略、状态和执行逻辑混在一起
- 副作用没有清晰边界地泄漏到系统各处
- 一个子系统重构迫使许多无关文件做附带修改

## 实现原则

这些原则来自成熟系统代码库，也来自本仓库使用的 provider CLI 形态。

### 编码不变量

让无效状态难以表示。

使用类型、schema、构造函数、状态机和验证边界，直接表达系统不变量。

不要依赖注释、命名或调用者纪律来维持代码可以编码的规则。

优先选择：

- typed state，而不是 stringly-typed status value
- 会验证持久对象的构造函数
- 围绕敏感状态转换的窄 public API
- 显式 capability 或 permission value，而不是环境式权限
- 当人和工具都依赖规则时，使用机器可读契约

### 让 public surface 保持普通

把内部复杂度隐藏在稳定、普通的接口之后。

实现越复杂，调用者面对的表面就越应该简单。

优先选择：

- 小入口
- 明显的命名
- 可预测的返回值
- 暴露真实选择的 feature flag 或配置，而不是内部机制
- 解释权衡的文档，而不是要求读者理解整个子系统

### 区分易用性和语义

便利不应模糊系统含义。

当 ergonomic helper 保留与低层 API 相同的失败模式、权威边界和数据模型时，它是好的。

避免这样的 helper：

- 跳过验证
- 隐藏状态所有权
- 吞掉有意义的错误
- 把策略决策伪装成格式化或传输细节
- 为同一操作创造第二条非正式路径

### 把错误视为系统信息

错误应该同时保留机器可操作的结构和对人有用的上下文。

当调用者只需要上下文和退出行为时，应用代码可以使用较宽泛的错误类型。

库、协议、存储和集成边界应该暴露足够精确的错误，使调用者可以做决定。

每条错误路径都应该回答：

1. 什么失败了？
2. 在哪里失败？
3. 是否可以恢复？
4. 调用者是否需要 typed distinction？
5. 调试时 operator 需要什么上下文？

### 约束危险能力

unsafe 操作、直接系统调用、全局状态、unchecked cast、原始并发原语、shell 执行、文件编辑和破坏性副作用，都必须隔离在小而经过审阅的边界之后。

这个边界应该说明：

- 哪个不变量使该操作有效
- 谁可以调用它
- 它可能修改什么状态
- 哪些测试、断言或运行时检查保护它

不要把特权操作散布在便利 helper 中。

### 为替换而设计

外部服务、存储 backend、模型 provider、runtime 和平台特定集成，都应该放在可替换 adapter 后面。

可替换不意味着假装所有 backend 完全相同。

它意味着：

- 共享策略位于 adapter 之上
- backend 特定行为留在 adapter 内部
- capability 差异是显式的
- 测试可以验证契约，而不依赖每一个真实 backend

### 在边界测试行为

对于风险存在于集成边界的代码，单元测试不够。

使用能捕捉真实失败模式的最便宜测试。

优先选择：

- 用小单元测试覆盖纯逻辑
- 用回归测试覆盖已修复 bug
- 用集成测试覆盖命令、API、存储和 adapter 行为
- 用 smoke test 覆盖终端、会话、路由和审批流程
- 只有在明确具备凭据和网络时，才运行 live provider smoke test

### 在代码附近解释权衡

当代码在性能、兼容性、可移植性、简单性、provider 行为或严格正确性之间做取舍时，把这个选择解释在实现附近。

解释应该说明为什么这里选择了这个方向，以及什么条件变化时可能需要重新评估。
